FAQ about the Data Loader

What is a Digestion?
  • The Data Loader is used to produce a Digestion which when executed loads data from input data files into ECoach
  • A Digestion targets a single ECoach variable called an MTS Characteristic and can take it's input from one column, or combine several columns, of data in the input file
  • A Digestion has various attributes associated with it which are configured using the Data Loader
  • These attributes are:
    1. Digestion ID (fixed)
    2. Digestion name
    3. Input data file
    4. List of columns from the input data file used to extract the value of a characteristic
    5. Id column of the input data file
    6. Method of combining data in the selected columns of an input data file
    7. MTS characteristic to populate with result
Is it complicated to create a Digestion?
  • No, it's not complicated
  • There are a couple ways to do it, however, so if you know the steps and how they work, and how to watch for problems, then creating a Digestion to load data into ECoach will go smoothly
What if my input file doesn't have an ID column matching the ECoach user names?
  • This is where a Map files comes in, a Map file provides a Mapping between IDs in the data file and ECoach user names, or unique names
What file formats are used as input to a Digestion?
  • The input data file is a CSV file formatted as follows:
    1. The data file should have exactly one header row at the top of the file
    2. The data file can have any number of data rows
    3. The data file has at least 1 ID column which is used to match rows of data to ECoach users
  • The ID Map file is a CSV file formatted as follows:
    1. The left column should match IDs present in the input data file, and the right column matches ECoach user names
    2. The IDs in the left column of the Map file must be unique
    3. The IDs in the right column can be redundant allowing the same Map file to be used for remapping the same student from two different external systems
    4. The IDs in the left column of the Map file should not match any of the ECoach user names to avoid privacy leaks, this violations is reported
Do I just follow the steps 1 through 4 in order?
  • For the most part yes, but there are times when an out of sequences step is handy, so you are allowed to go back and forth between steps.
  • We will start by looking at the simplest 1,2,3,4 step sequence and then explore when to use an out of sequence step
  • While the four steps are different there are a few similarities between them
    1. The submit button is always in the same position at the top of the page
    2. Each step allows for you to make some choices in setting variables to configure the action for that step
    3. The value of the variables you can edit on a page is always displayed in the upper left one line below the submit button
    4. The information that a step reports after running is laid out in different areas of the page depending on the step, however these report always have a grey background
What happens when I click submit?
  • The details of what happens depend on the step, but in each case your request goes to the server, and the server does some work and returns the result
  • The first three steps will do their action on the server, and return you to the same page updating it with the result report
  • The 4th step, when committed actually runs the Digestion, archives it, creates a new Digestion, and redirects to the archive step upon completion
  • Since steps 2, 3, and 4 require an input data file to execute they will redirect to step 1 if non has been selected
When is clicking submit necessary?
  • Each step should be submitted once and reviewed, especially when cloning is used.
  • In order to actually *DO* a step, you will always have to press the submit button. For example, if you cloned a Digestion that has column 33 selected, and then changed the input file to correct some mistake, you could run into trouble if the submit button were not used to resubmit step 3. While column 33 doesn't appear selected, that's only because there's no column 33 to display in the new data file. Without re-submitting step 3 column 33 would still be included as part of the Digestion attributes, causing problems since it doesn't exist in the new file.
How is the archive used?
  • The archive is used to copy or clone a previous Digestion into the current working Digestion
  • Once an archived Digestion is copied or cloned it can be explored for review
  • A cloned Digestion can be modified as desired and then rerun
  • Being able to clone a previous Digestion saves time when multiple characteristics are being extracted from the same data file, or when the same characteristic is being extracted from various data files
Which function or method should I choose?
  • avg1([cols])
    1. Finds % of total points achieved from all assignments
    2. Assignment contribution varies as it's determined by the sum of points achieved divided by the total points possible from all assignments
    3. Use with numeric input
  • avg2([cols])
    1. Finds the average number of points per assignment over all assignments
    2. Value never converted to percent, simply the average of all the numbers
    3. Use with numeric input
  • avg3([cols])
    1. Treats each assignement as same % of grade regardless of points
    2. Assignemnt contribution constant no matter how many points
    3. Use with numeric input
  • pick(col)
    1. Use the value directly from the column selected
    2. Pick the first column if multiple are selected
    3. Use with any kind of input
  • The other options are not implemented yet!

Data Loader Usage Patterns

Ordinary sequence:
Step 1. Upload File
- Load a properly formatted input data file
- Optionally load a properly formatted ID Map file
Step 2. Review File
- Select the appropriate ID column from the data file
- Review the reports to verify files meet validity requirements and your expectations
Step 3. Digest Data
- Select the MTS characteristic targeted by the Digestion
- Select the data columns to combine when producing the value for the mts characteristic
- Select the method of combining data columns
Step 4. Commit
- Name of the Digestion before committing
Cloning sequence:
Archive
- Clone an archived Digestion into the current working Digestion
Step 1. Upload File
- Review/modify the selected Data and Map files
Step 2. Review File
- Review/modify the selected ID column
Step 3. Digest Data
- Review/modify the MTS characteristic being targeted
- Review/modify the data columns selected
- Review/modify the selected method of combining data columns
Step 4. Commit
- Edit the name of the Digestion before committing
Reviewing sequence:
Archive
- Clone an archived Digestion into the current working Digestion
Steps 1-4. Review
- Explore the cloned Digestion by reviewing steps in any order
